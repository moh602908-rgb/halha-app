# دلّني AI

**الإصدار الحالي: v2.0.0**

حلول عملية للحياة اليومية — تطبيق ويب (PWA) يقدّم أدلة خطوة بخطوة لحل أكثر مشاكل الحياة اليومية شيوعًا، بالعربية، ويعمل بالكامل دون إنترنت بعد أول زيارة، مع بحث ذكي يفهم الأخطاء الإملائية والمرادفات.

## نطاق هذا الإصدار (Version 1.0)
- **24 مجالًا متوازنًا**، كل مجال يحتوي **7 أدلة كاملة بالضبط** (168 دليلاً إجمالاً)، بدون أي دليل "قيد الكتابة" وبدون أي عنوان مكرر.
- تمت مراجعة كل المحتوى للتأكد من خلوّه من أي استشارة طبية أو قانونية أو نفسية متخصصة، أو محتوى حساس (حمل/ولادة، علاقات زوجية/حميمية)، ليتوافق مع سياسات متاجر التطبيقات مثل Google Play.
- **بحث محلي ذكي بالكامل** (بدون إنترنت أو أي خدمة خارجية): يتجاهل الهمزات/التاء المربوطة/الألف المقصورة/التشكيل، يتسامح مع خطأ إملائي بسيط، يفهم مرادفات شائعة (جوال/هاتف، فلوس/مال...)، ويعرض أقرب النتائج بدل "لا توجد نتائج" عند عدم وجود تطابق تام.
- نافذة **"مساعد دلّني"**: وصول سريع لأكثر الأدلة استخدامًا + نفس محرك البحث الذكي، بدون ذكاء اصطناعي في هذه المرحلة (سيُضاف في المرحلة الثانية عبر خادم خاص، وليس مفتاح API مكشوف في المتصفح).
- يعمل كتطبيق PWA قابل للتثبيت على أندرويد وiOS والحاسوب، بأيقونات PNG قياسية (192/512 + نسخة maskable)، ودعم تصفح دون إنترنت بعد أول زيارة.

## سجل مراجعة v1.2.0 (آخر مراجعة قبل الاعتماد)
- **إصلاح جذري**: مربع البحث الرئيسي كان يفقد التركيز (Focus) بعد كل حرف، لأن الصفحة كانت تعيد إنشاء عنصر `<input>` بالكامل مع كل نتيجة بحث. أُصلح عبر فصل حقل البحث عن منطقة النتائج بحيث لا يُعاد إنشاء الحقل أبدًا أثناء الكتابة.
- **بحث ذكي جديد بالكامل**: تطبيع الكتابة العربية (أ/إ/آ ← ا، ة ← ه، ى ← ي، إزالة التشكيل)، مرادفات شائعة، بحث تقريبي يتسامح مع خطأ إملائي، وعرض "أقرب النتائج" بدل رسالة فارغة. يعمل بالكامل محليًا دون إنترنت، وسرعته دون الميلي ثانية الواحدة حتى مع 168 دليلاً.
- إضافة أيقونات PNG قياسية (192 / 512 / نسخة maskable / نسخة iOS) لضمان أفضل توافق تثبيت على كل الأجهزة.
- إصلاح تكرار أيقونة 🌱 بين مجالين (أصبحت الزراعة 🪴).
- مراجعة شاملة أكدت: 24 مجالًا × 7 أدلة بالضبط = 168، بلا تكرار، بلا محتوى حساس، الكل بنفس الأسلوب.
- اختُبر التطبيق فعليًا بمتصفح Chromium حقيقي (ليس تخمينًا): البحث، التنقل، نافذة المساعد، العمل دون إنترنت، ودورة نشر تحديث كاملة — كل الاختبارات نجحت.

## الملفات
- `index.html` — الصفحة الرئيسية وهيكل التطبيق
- `style.css` — التصميم والألوان
- `app.js` — بيانات الأدلة (24 مجالًا متوازنًا)، محرك البحث الذكي، ومنطق التنقل ونافذة المساعد
- `manifest.json` — إعدادات PWA (قابلية التثبيت)
- `icon.svg` — الأيقونة الأساسية (متجهية، تُستخدم كخيار احتياطي عالي الجودة)
- `icon-180.png` — أيقونة شاشة الهاتف الرئيسية لأجهزة iOS تحديدًا (Apple Touch Icon)
- `icon-192.png` / `icon-512.png` — أيقونات PWA القياسية لأندرويد والحاسوب
- `icon-512-maskable.png` — نسخة مخصصة لأندرويد بهامش أمان كافٍ حتى لا تُقص عند تطبيق قصّ الشكل التلقائي للأيقونات (Adaptive Icons)
- `icon-maskable.svg` — الملف المصدر (متجهي) لتوليد `icon-512-maskable.png`؛ غير مستخدم مباشرة وقت تشغيل التطبيق، محفوظ فقط لتسهيل تعديل الأيقونة مستقبلاً
- `sw.js` — العمل دون إنترنت + التحكم بتحديث النسخة المعروضة للمستخدم
- `vercel.json` — يمنع تخزين الملفات مؤقتًا على مستوى الاستضافة/الشبكة، فيضمن وصول التحديثات فورًا
- `api/ask.js` — بوابة خلفية آمنة (Edge Function) تحمي مفتاح الذكاء الاصطناعي — راجع قسم "المرحلة الثانية" بالأسفل
- `lib/config.js` — كل الإعدادات القابلة للتغيير (الحدود اليومية، أسماء النماذج) في مكان واحد
- `lib/providers.js` — Provider Manager: يعزل تفاصيل كل مزوّد ذكاء اصطناعي في مكان واحد
- `lib/ratelimit.js` — عدّاد استخدام يومي موقّع رقميًا (بدون قاعدة بيانات)
- `lib/prompt.js` — بناء التوجيه (System Prompt) وتنظيف مدخلات المستخدم
- `package.json` — تعريف بسيط للمشروع (بدون أي حزم npm خارجية) لجعل `api/ask.js` وحدة ES Module
- `.env.example` — توثيق متغيرات البيئة المطلوبة (بدون قيم حقيقية — آمن للرفع على Git)
- `.gitignore` — يمنع رفع أي ملف أسرار محلي بالخطأ

## ⚠️ تنبيه مهم جدًا لأي تحديث مستقبلي
في كل مرة تُحدَّث فيها ملفات المشروع ثم تُرفع من جديد، **يجب تغيير رقم `APP_VERSION` في أعلى ملف `sw.js`** (مثلاً من `v2.0.0` إلى `v2.1.0`).
بدون هذه الخطوة، سيستمر متصفح كل مستخدم بعرض **النسخة القديمة المحفوظة عنده**، حتى لو نجح الرفع على GitHub وVercel بنجاح تام.

## 🐛 إصلاح مشكلة "النسخة القديمة تظهر رغم الرفع" (تمّ حلها في v1.1.0)
السبب الجذري الذي عُثر عليه: داخل `sw.js`، كان استدعاء `fetch(req)` بدون تحديد وضع الكاش يستخدم الوضع الافتراضي، والذي يسمح للمتصفح بالإجابة من ذاكرته الداخلية (HTTP Cache) دون الاتصال الفعلي بالشبكة — رغم أن نية الكود كانت "الشبكة أولًا دائمًا". هذا كان يُبطل التحديث حتى مع تغيير `APP_VERSION`، وأحيانًا حتى في التصفح المتخفي إن كانت هناك طبقة تخزين مؤقت وسيطة (كخادم Vercel نفسه) تُرجع نسخة قديمة.
تم إصلاح هذا نهائيًا عبر:
1. **`sw.js`**: كل الطلبات المارة عبر الـ Service Worker تستخدم الآن `{ cache: "no-store" }` صراحة، فتُجبر على الاتصال الفعلي بالخادم دائمًا (لا اعتماد على أي ذاكرة HTTP وسيطة).
2. **`index.html`**: تسجيل الـ Service Worker يستخدم الآن `updateViaCache: "none"` (يمنع تخزين ملف `sw.js` نفسه مؤقتًا)، ويطلب فحص تحديث فوري (`reg.update()`) عند كل تحميل صفحة، ويعيد تحميل الصفحة تلقائيًا مرة واحدة فور تفعيل نسخة جديدة — فلا يحتاج المستخدم لتحديث يدوي أو مسح بيانات الموقع أبدًا.
3. **`vercel.json`** (ملف جديد): يضبط رؤوس `Cache-Control: no-cache, no-store, must-revalidate` على كل الملفات عند النشر على Vercel، لمنع أي تخزين مؤقت على مستوى الخادم أو الشبكة (CDN) نفسه. العمل دون إنترنت لا يتأثر بهذا إطلاقًا، لأنه يعتمد بالكامل على `Cache Storage` الخاصة بالـ Service Worker، وهي منفصلة تمامًا عن ذاكرة HTTP Cache العادية للمتصفح.

**ملاحظة لمرة واحدة فقط:** المستخدمون الذين زاروا الموقع قبل هذا الإصلاح لديهم نسخة قديمة من `sw.js` نفسه ما تزال تعمل مؤقتًا في متصفحهم. زيارتهم التالية للموقع (حتى دون أي إجراء يدوي منهم) ستكتشف تلقائيًا وجود `sw.js` جديد، وتحدّث نفسها، وتُعيد تحميل الصفحة تلقائيًا مرة واحدة — دون أي تدخل من المستخدم.

## لا يحتاج أي تثبيت
هذا مشروع HTML/CSS/JavaScript خام بدون أي أدوات بناء (لا Node.js ولا npm مطلوبة لتشغيله أو نشره). يكفي رفع الملفات كما هي إلى الاستضافة.

## سياسة المحتوى (مهم عند إضافة أدلة جديدة لاحقًا)
لتفادي أي مشكلة مستقبلية مع متاجر التطبيقات، تجنّب دائمًا أي دليل جديد يتضمن:
- تشخيصًا طبيًا أو وصف جرعات/علاجات دوائية محددة.
- محتوى عن الحمل والولادة والرعاية الطبية للمولود.
- محتوى عن العلاقات الزوجية أو الحميمية أو الصحة الجنسية.
- استشارة قانونية أو نفسية متخصصة (تحويل المستخدم لمختص عند الحاجة مقبول، لكن ليس تقديم الاستشارة نفسها).
الأدلة الحالية كلها إما مهارات عملية عامة، أو معلومات سلامة وإسعافات أولية أساسية (لا تُعتبر استشارة طبية متخصصة)، أو توعية عامة مع توجيه واضح لمختص عند الحاجة.

## إضافة دليل أو مجال جديد لاحقًا
أضف عنصرًا جديدًا داخل مصفوفة `guides` الخاصة بالمجال المناسب في `app.js`، بنفس شكل الأمثلة الجاهزة (`title` + `ready: true` + `steps` + `note` اختياري). حافظ على 7 أدلة لكل مجال للحفاظ على التوازن. لإضافة مجال جديد بالكامل، أضف كائنًا جديدًا لمصفوفة `DOMAINS` بنفس البنية (`id` فريد، `name`، `icon`، `guides`) بـ7 أدلة كاملة منذ البداية.

## 🤖 المرحلة الثانية: الذكاء الاصطناعي (v2.0.0)

### الفكرة العامة
التطبيق يبقى كما هو (PWA بدون خادم تقليدي)، لكن أُضيفت **دالة خادم واحدة صغيرة** (`api/ask.js`) تعمل على Vercel كـ Edge Function. هذه الدالة هي الوسيط الوحيد بين التطبيق ومزوّد الذكاء الاصطناعي، وهي التي تحمل المفتاح السرّي — المتصفح لا يراه أبدًا.

الترتيب الملتزم به دائمًا في الواجهة (`app.js`): **البحث المحلي في الأدلة أولاً** (فوري، مجاني، يعمل دون إنترنت) → إن لم يجد المستخدم ما يكفي، يضغط بنفسه على زر **"اسأل الذكاء الاصطناعي"** → عندها فقط يُرسل الطلب. الذكاء الاصطناعي ليس مصدر الإجابة الأول أبدًا ولا يُستدعى تلقائيًا.

### التفعيل (خطوتان فقط على Vercel)
1. من **Vercel Dashboard → Project → Settings → Environment Variables** أضف:
   - `AI_API_KEY` — مفتاح Anthropic API الحقيقي (من console.anthropic.com).
   - `RATE_LIMIT_SECRET` — نص عشوائي طويل وفريد (راجع `.env.example` لطريقة توليده).
   - (اختياري) `FREE_DAILY_LIMIT`، `DEFAULT_MODEL`، `PREMIUM_DAILY_LIMIT`، `PREMIUM_MODEL`، `AI_PROVIDER` — إن أردت تغيير القيم الافتراضية دون لمس الكود.
2. أعد النشر (Redeploy). لا حاجة لأي تعديل آخر — `api/ask.js` يعمل تلقائيًا لأنه داخل مجلد `api/`.

بدون هاتين الخطوتين، يعمل التطبيق تمامًا كما في الإصدار v1 (بحث محلي فقط)، وزر "اسأل الذكاء الاصطناعي" يعرض رسالة خطأ لطيفة بدل الانهيار.

### الأمان — لماذا هذا التصميم "الأفضل ممكن" لتطبيق Frontend فقط
| المتطلب | كيف تحقق |
|---|---|
| المفتاح لا يظهر في JS/HTML/الملفات المنشورة | يعيش فقط في متغير بيئة على خادم Vercel، لا وجود له في أي ملف بالمشروع |
| المفتاح لا يظهر في Developer Tools/Network | المتصفح لا يتصل بمزوّد الذكاء الاصطناعي مباشرة إطلاقًا؛ يتصل فقط بـ `/api/ask` على نفس نطاق موقعك |
| حماية الحد المجاني من التلاعب | كوكي HttpOnly موقّعة بـ HMAC-SHA256 بمفتاح سرّي مستقل (`RATE_LIMIT_SECRET`)؛ أي تعديل يدوي على العدّاد يُبطل التوقيع ويُعاد العدّ من صفر — تم اختبار هذا فعليًا (راجع "الاختبارات" بالأسفل) |
| منع مواقع أخرى من استخدام مفتاحك | `api/ask.js` يرفض أي طلب من نطاق (Origin) مختلف عن نطاق موقعك نفسه |
| خيار "مفتاحك الخاص" | زر ⚙️ في نافذة المساعد يتيح للمستخدم إدخال مفتاح Anthropic خاص به، يُحفظ في `localStorage` بمتصفحه فقط ولا يُرسل لخوادمنا إطلاقًا؛ الاتصال حينها مباشر من متصفحه لمزوّد الخدمة بمفتاحه هو (نمط BYOK القياسي المعتمد رسميًا من Anthropic عبر رأس `anthropic-dangerous-direct-browser-access`) |

### إدارة التكلفة (كل هذا يعمل قبل أي اتصال فعلي بالذكاء الاصطناعي)
1. **البحث المحلي أولاً** دائمًا — لا تكلفة إطلاقًا إن وجد المستخدم ضالته في الأدلة.
2. **سياق مختصر فقط**: عند سؤال الذكاء الاصطناعي، تُرسل أفضل 3 أدلة محلية ذات صلة (وليس الأدلة الـ168 كلها) كمرجع للنموذج — يحسّن الجودة ويقلل الحجم معًا.
3. **ذاكرة محادثة محدودة**: آخر 6 رسائل فقط تُرسل كسياق، لا المحادثة كاملة.
4. **تفادي تكرار نفس السؤال**: كل سؤال وإجابته يُخزَّنان في `sessionStorage`؛ نفس السؤال (بعد تطبيع الكتابة العربية) لا يُرسل للخادم مرتين في نفس الجلسة.
5. **نموذج اقتصادي افتراضيًا**: `claude-haiku-4-5-20251001` — الأسرع والأرخص، مناسب تمامًا لأسئلة الحياة اليومية.
6. **حد يومي قابل للتعديل** (`FREE_DAILY_LIMIT`، افتراضيًا 10) يمنع أي استخدام غير متوقع لمفتاحك.

### Provider Manager — التوسع لمزوّدين آخرين
`lib/providers.js` يعزل كل تفاصيل مزوّد الذكاء الاصطناعي عن باقي المشروع. لإضافة مزوّد جديد (Gemini، OpenAI، أو أي مزوّد آخر):
1. اكتب دالة بنفس شكل `callClaude(...)` الموجودة في الملف (نفس المدخلات، تُرجع نص الإجابة).
2. سجّلها في كائن `PROVIDERS`.
3. غيّر متغير البيئة `AI_PROVIDER` إلى اسمها.

لا حاجة لتعديل `api/ask.js` ولا `app.js` ولا أي ملف آخر — هذا هو معنى "طبقة مستقلة عن المزوّد".

### الإعدادات المركزية (`lib/config.js`)
كل رقم أو اسم نموذج قابل للتعديل من مكان واحد فقط، أو عبر متغيرات البيئة دون إعادة نشر الكود:

| المتغير | القيمة الافتراضية | الوصف |
|---|---|---|
| `FREE_DAILY_LIMIT` | 10 | الحد اليومي للمستخدم المجاني |
| `PREMIUM_DAILY_LIMIT` | 50 | جاهز لخطة مميزة مستقبلية (غير مفعّلة بعد) |
| `DEFAULT_MODEL` | `claude-haiku-4-5-20251001` | نموذج الخطة المجانية |
| `PREMIUM_MODEL` | `claude-sonnet-5` | جاهز لخطة مميزة مستقبلية |
| `AI_PROVIDER` | `claude` | المزوّد النشط حاليًا |

دالة `resolvePlan()` في نفس الملف هي **نقطة التوسع الوحيدة** لتفعيل خطط المستخدمين لاحقًا (تسجيل دخول + قاعدة بيانات): عدّلها لتقرأ حالة المستخدم الفعلية بدل إرجاع "مجاني" دائمًا، وسيتبعها تلقائيًا كل شيء آخر (الحد، النموذج) دون تعديل أي ملف آخر.

### قابلية النشر عبر أي منصة
طبقة الذكاء الاصطناعي بأكملها (`api/` و`lib/`) لا تفترض أنك على Vercel تحديدًا في منطقها الداخلي — تعتمد فقط على Fetch API وWeb Crypto القياسيين، وقابلة للنقل لأي استضافة Edge/Serverless أخرى (Cloudflare Workers، Netlify Functions...) بأقل تعديل. أما التطبيق نفسه (الواجهة)، فبما أنه PWA عادي، فإن تغليفه لاحقًا كتطبيق على Google Play (عبر Trusted Web Activity/Bubblewrap) أو أي متجر آخر يعمل دون أي تعديل: التطبيق المُغلَّف يبقى يحمّل نفس الموقع الحي، وطلبات `/api/ask` تذهب لنفس الخادم كما في المتصفح تمامًا.

### الاستقرار والذاكرة
- ذاكرة المحادثة داخل الجلسة الحالية فقط (`sessionStorage`) — تُمسح تلقائيًا بإغلاق التبويب، وليست حسابًا دائمًا. لا حسابات مستخدمين ولا قاعدة بيانات ولا مزامنة سحابية في هذه المرحلة، تمامًا كما طُلب.
- زر "محادثة جديدة" داخل نافذة المساعد يمسح الذاكرة يدويًا قبل ذلك أيضًا.
- أي خطأ اتصال (لا إنترنت، الخادم غير مُعدّ بعد، تجاوز الحد اليومي) يُعرض كرسالة عربية واضحة داخل المحادثة نفسها، ولا يكسر باقي التطبيق أبدًا — البحث المحلي والتصفح يستمران بالعمل بشكل طبيعي دائمًا بغض النظر عن حالة الذكاء الاصطناعي.

### ✅ ثغرات حقيقية اكتُشفت وأُصلحت في هذه الجولة
1. **غياب حد لحجم الطلب (DoS بسيط)**: كان بالإمكان إرسال حمولة ضخمة (اختُبر فعليًا بـ 2 مليون حرف) قبل أي تحقق. **الإصلاح**: رفض أي طلب أكبر من 100 كيلوبايت فورًا (413) قبل قراءته بالكامل، بالتحقق من `Content-Length` أولاً ثم من الحجم الفعلي بعد القراءة كطبقة ثانية.
2. **غياب التحقق من Content-Type**: كان الخادم يقبل أي نوع محتوى طالما جسم الطلب JSON صالح. **الإصلاح**: رفض أي طلب لا يحمل `Content-Type: application/json` صراحة (415).
3. **عدم وجود مهلة زمنية (Timeout) على الاتصال بمزوّد الذكاء الاصطناعي**: طلب معلّق كان يمكن أن يستهلك موارد الدالة بلا حد. **الإصلاح**: `AbortController` بمهلة 25 ثانية قابلة للتعديل.
4. **عدم وجود أي تسجيل لمحاولات الإساءة**: لم يكن هناك أي أثر يمكن مراجعته لاحقًا. **الإصلاح**: تسجيل بسيط (`console.warn`) لكل محاولة Origin مرفوض، تجاوز حد، حمولة كبيرة، أو خطأ من المزوّد — يظهر في Vercel Logs مباشرة بدون أي بنية تحتية إضافية.

### ⚠️ قيود معروفة لم تُخفَ (كما طُلب صراحة عدم التستر عليها)

**1) حماية الحد اليومي قائمة على كوكي من جهة العميل — وهذا له حد بنيوي حقيقي**

اختبرتُ هذا فعليًا بطريقتين مختلفتين وأثبتّه بأرقام حقيقية:
- **تجاوز كامل عبر سكربت لا يرسل الكوكي**: نفّذتُ 15 طلبًا متتاليًا دون إرسال أي كوكي في أي منها، ونجحت كلها الـ15 رغم أن الحد المعلن 10. **السبب**: الحد بأكمله يعتمد على أن العميل "يرسل بأمانة" الكوكي التي أعطيناه إياها في المرة السابقة. متصفح حقيقي يفعل هذا تلقائيًا دون أن يفكر المستخدم في الأمر، لكن سكربت هجومي (curl، أو كود بايثون) يمكنه ببساطة عدم إرسالها إطلاقًا في كل مرة، فيبدو كل طلب وكأنه "أول طلب" من متصفح جديد.
- **Race Condition عند التزامن**: أرسلتُ 8 طلبات متوازية تمامًا (بنفس نقطة البداية) عبر `Promise.all` — نجحت كلها الـ8 لأن كل طلب قرأ قيمة العدّاد **قبل** أن يُصدر أي طلب آخر كوكيه الجديدة. هذا نتيجة حتمية لأي عدّاد بلا حالة مشتركة (Shared State) بين الطلبات المتزامنة.

**التأثير الحقيقي على المستخدم العادي**: لا يوجد أي تأثير سلبي إطلاقًا. مستخدم عادي يفتح التطبيق ويكتب أسئلته من متصفحه بشكل طبيعي لن يواجه أو يستغل أيًا من هذا — الحماية تعمل بشكل صحيح تمامًا في الاستخدام الطبيعي (مُثبَت باختبارات القسم السابق: 10 طلبات تُقبل، الحادي عشر يُرفض، والتلاعب اليدوي بالكوكي من أدوات المطوّر يُرفض ويُعاد العدّ من صفر).

**متى تصبح هذه الثغرة مؤثرة فعليًا**: فقط إذا استهدف شخصٌ التطبيق تحديدًا بسكربت آلي بنيّة استنزاف ميزانية مفتاح API عمدًا. هذا سيناريو "مهاجم مصمم" لا "مستخدم عادي يتحايل قليلًا".

**لماذا لم أحلّها بالكامل**: الحل الوحيد الحقيقي (عدّاد ذري مشترك بين كل الطلبات) يتطلب حالة مركزية خارج الطلب نفسه — أي قاعدة بيانات أو مخزن مفاتيح/قيم (Redis/KV)، وقد طُلب مني صراحة تجنّب ذلك في هذه المرحلة. بنيت النظام بحيث تكون هذه الترقية سهلة لاحقًا (راجع "الترقية المقترحة" أدناه) دون إعادة كتابة أي شيء آخر.

**2) فحص Origin يحمي من مواقع أخرى، لا من سكربت مباشر**

رأس `Origin` يُرسله المتصفح تلقائيًا وبصدق في كل طلب Fetch عابر للنطاقات، وهذا يحمي فعليًا من موقع آخر يحاول استخدام الـ endpoint من داخل صفحة ويب في متصفح شخص آخر (وهذا مُثبَت: طلب بـ`Origin: https://attacker-site.example` يُرفض 403 كما هو متوقع). لكن أداة سطر أوامر (curl، Python) يمكنها ببساطة **تلفيق** أي قيمة Origin تريدها، بما فيها نطاقك الحقيقي بالضبط — فحص Origin هنا خط دفاع ضد إساءة الاستخدام من متصفحات أخرى، وليس مصادقة (Authentication) حقيقية ضد مهاجم مباشر.

### 🛡️ الخط الدفاعي الحقيقي الأخير (الأهم عمليًا)
بما أن الحماية البرمجية البحتة (بدون قاعدة بيانات) لها حدود بنيوية كما وُثِّق أعلاه، **الحماية الفعلية النهائية لميزانيتك هي إعداد سقف إنفاق (Spending Limit / Budget Alert) داخل حساب Anthropic Console نفسه**. هذا ليس اختياريًا — أنصح بشدة بضبطه **قبل** نشر المفتاح الحقيقي في بيئة الإنتاج، بغض النظر عن أي حماية برمجية أخرى. هذا يضمن أنه حتى في أسوأ سيناريو استغلال متعمّد، التكلفة القصوى محدودة وتحت سيطرتك الكاملة دائمًا.

### 🔼 الترقية المقترحة إن أصبح الاستغلال المتعمّد مشكلة فعلية
أضف مخزن مفاتيح/قيم خفيف الوزن (**Vercel KV** أو **Upstash Redis** — كلاهما له مستوى مجاني كافٍ لهذا الحجم من الاستخدام) لجعل عدّاد الاستخدام ذريًا (Atomic) ومشتركًا فعليًا بين كل الطلبات المتزامنة، مفتاحًا بعنوان IP بدل الاعتماد على كوكي العميل وحدها. بنية `lib/ratelimit.js` الحالية معزولة تمامًا خلف دالة واحدة (`checkAndPrepareUsage`) بالضبط لهذا الغرض — الترقية تعني استبدال محتوى هذه الدالة فقط بنسخة تستخدم `INCR` الذرية لـ Redis، دون تعديل `api/ask.js` ولا أي ملف آخر بالمشروع إطلاقًا.

### حماية حسابات مالك المشروع (فحصتها تحديدًا بناءً على طلب صريح)
- المشروع يحتوي بيانات اعتماد لمزوّد واحد فقط (`AI_API_KEY` لـ Anthropic) — لا يوجد أي مفتاح أو بيانات اعتماد أخرى (لا ChatGPT، لا Gemini فعليًا، رغم وجود "أطر جاهزة" فارغة لهما في Provider Manager ترمي خطأ فورًا إن استُدعيت).
- تأكدتُ بالاختبار الفعلي (قسم "محاولة تجاوز Provider Manager" أعلاه) أن العميل لا يستطيع حقن حقل `provider` أو `model` أو `apiKey` في الطلب لإجبار الخادم على استخدام مزوّد أو نموذج مختلف عمّا حدده الخادم نفسه — كل هذه الحقول من العميل تُتجاهل بصمت والخادم وحده يقرر.
- ميزة "مفتاحك الخاص" (BYOK) مصمَّمة بعزل كامل: تتصل من متصفح المستخدم مباشرة بـ Anthropic بمفتاحه هو، ولا تمر عبر خادمنا إطلاقًا ولا تستطيع الوصول لمفتاح المالك بأي شكل بنيويًا (المفتاحان في متغيرين منفصلين تمامًا في كودين منفصلين تمامًا: مفتاح المالك في `process.env` على الخادم، ومفتاح المستخدم في `localStorage` بمتصفحه هو).

### 🚨 خطة الطوارئ عند الاشتباه بتسرّب المفتاح
1. **ألغِ المفتاح فورًا** من Anthropic Console (Settings → API Keys → Revoke) — هذه أهم خطوة وأولها، قبل أي شيء آخر.
2. **أنشئ مفتاحًا جديدًا** من نفس الصفحة.
3. **حدّث `AI_API_KEY`** في Vercel Dashboard → Project → Settings → Environment Variables بالقيمة الجديدة.
4. **أعد النشر (Redeploy)** حتى تسري القيمة الجديدة فعليًا (تغيير متغير بيئة على Vercel لا يُطبَّق على نشر قائم بالفعل تلقائيًا).
5. **(احتياطي) غيّر `RATE_LIMIT_SECRET` أيضًا** إن اشتبهت بتسرّبه هو أيضًا — هذا يُبطل كل كوكيز الاستخدام الحالية لدى كل المستخدمين ويبدأ العدّ من جديد للجميع، وهو تأثير جانبي بسيط ومقبول.
6. راجع **Vercel Logs** (Project → Logs) بحثًا عن نمط غير طبيعي من رسائل `[dallini-security]` (المُضافة في هذه المراجعة تحديدًا لهذا الغرض) قبل التسريب وبعده لفهم حجم وطريقة الاستغلال إن وُجد.
7. تحقق من **سجل الفوترة (Billing)** في Anthropic Console للتأكد من عدم وجود استهلاك غير متوقع، واضبط سقف إنفاق فوري إن لم يكن مضبوطًا أصلاً.

### دعم اللهجات العربية
حسّنتُ التوجيه (System Prompt) في `lib/prompt.js` بتعليمة صريحة: افهم أي لهجة عربية يكتب بها المستخدم (خليجية، مصرية، شامية، جزائرية/مغربية/تونسية، أو غيرها)، لكن أجب دائمًا بالفصحى المبسّطة الموحّدة. **ملاحظة أمانة مهمة**: لا يوجد في هذه البيئة اتصال فعلي بنموذج Claude حقيقي لاختبار فهم اللهجات حيًا (لا يوجد مفتاح API حقيقي متاح للاختبار المباشر هنا) — التحسين الذي نُفِّذ هو تصميم التوجيه بشكل صريح يوجّه النموذج لهذا السلوك، وهو أسلوب معياري وفعّال بشدة مع نماذج Claude الحديثة في التعامل مع اللهجات العربية المختلفة، لكنه لم يُختبَر بمكالمة API حقيقية داخل هذه المراجعة تحديدًا. يُنصح بتجربة عيّنة أسئلة من كل لهجة فور تفعيل المفتاح الحقيقي للتأكد قبل الإطلاق الكامل للعامة.
